## Setup Files

**Location:** `src/test/setup.ts`

**Contents:**
```typescript
import "@testing-library/jest-dom";

Object.defineProperty(window, "matchMedia", {
  writable: true,
  value: (query: string) => ({
    matches: false,
    media: query,
    onchange: null,
    addListener: () => {},
    removeListener: () => {},
    addEventListener: () => {},
    removeEventListener: () => {},
    dispatchEvent: () => {},
  }),
});
```

**Purposes:**
- Imports jest-dom matchers for DOM testing
- Mocks `window.matchMedia()` to support media query testing (used by responsive hooks like `useIsMobile`)
- Runs before all tests

## Mocking

**Framework:** Vitest built-in mocking

**What to Mock:**
- Browser APIs (e.g., `window.matchMedia` - already mocked in setup)
- External dependencies in integration tests
- Radix UI and other third-party components when testing in isolation

**What NOT to Mock:**
- React's built-in hooks (useState, useEffect, useContext)
- Internal utility functions unless testing error boundaries
- UI component library components (use real components from `@/components/ui`)

**Pattern for DOM APIs:**
```typescript
// Already configured in setup.ts for media queries
Object.defineProperty(window, "matchMedia", {
  writable: true,
  value: (query: string) => ({
    matches: false,
    media: query,
    // ... other properties
  }),
});
```

## Test Types

**Unit Tests:**
- Scope: Individual components and utility functions
- Approach: Test component behavior, props handling, event handlers
- Setup: Use `@testing-library/react` render function
- Example location: Tests for individual components like `Button.tsx`, `HeroSection.tsx`

**Integration Tests:**
- Scope: Multiple components working together (e.g., Form with FormField)
- Approach: Test interaction between components and state management
- Setup: Use `@testing-library/react` with user event simulation
- Not yet implemented in codebase

**E2E Tests:**
- Framework: Not used
- No E2E testing setup (no Cypress, Playwright, or WebDriver)
- Could be added for critical user journeys (sign-up, navigation flows)

## Common Test Patterns

**Basic Component Test Structure:**
```typescript
import { render, screen } from "@testing-library/react";
import { Button } from "@/components/ui/button";

describe("Button", () => {
  it("should render with children", () => {
    render(<Button>Click me</Button>);
    expect(screen.getByText("Click me")).toBeInTheDocument();
  });

  it("should handle click events", () => {
    const handleClick = vi.fn();
    render(<Button onClick={handleClick}>Click</Button>);

    screen.getByText("Click").click();
    expect(handleClick).toHaveBeenCalled();
  });
});
```

**Async Testing Pattern:**
```typescript
import { render, screen, waitFor } from "@testing-library/react";

it("should load data", async () => {
  render(<ComponentThatFetches />);

  await waitFor(() => {
    expect(screen.getByText("Loaded Data")).toBeInTheDocument();
  });
});
```

**Error Testing Pattern:**
```typescript
import { render, screen } from "@testing-library/react";

it("should throw error when hook used outside context", () => {
  // Suppress error output for this test
  const spy = vi.spyOn(console, "error").mockImplementation(() => {});

  expect(() => {
    render(<HookThatRequiresContext />);
  }).toThrow("useFormField should be used within <FormField>");

  spy.mockRestore();
});
```
